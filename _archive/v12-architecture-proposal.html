SLASHR Sales System v12 — Architecture cible

Integration des forces de claude-seo dans le systeme de closing

0. Principes directeurs de la refonte

Garder ce qui marche

Les 3 passes restent sequentielles (Data → Narration → HTML). Les checkpoints closers sont un avantage competitif. Les 54 regles de validation sont uniques. Le cache/replay est deja mature.

Paralleliser A L'INTERIEUR de chaque passe

La collecte (Pass 1) et la generation des onglets (Pass 3) deviennent paralleles via des subagents, comme claude-seo le fait pour son audit. Le flux global reste sequentiel, mais chaque passe est 3-5x plus rapide.

Modulariser les sources de donnees

Chaque source (Pipedrive, Drive, DataForSEO, GSC, Google Ads) devient une extension autonome avec son agent dedie. Ajouter une source = ajouter un dossier, pas modifier 5 fichiers.

Ajouter des commandes d'analyse rapide

Comme claude-seo a /seo page et /seo technical en plus de /seo audit, SLASHR gagne /audit (diagnostic rapide) et /benchmark (analyse concurrentielle standalone).

Ponderation SLASHR (orientee closing, pas audit technique)

CategoriePoidsJustification (vs claude-seo)
Potentiel de marche (volumes, gaps)30%claude-seo: 0%. C'est la raison d'etre du closing : prouver l'opportunite.
Technical SEO20%claude-seo: 22%. Similaire, reste un fondamental.
Content & E-E-A-T20%claude-seo: 23%. Crucial pour la recommandation.
Benchmark concurrentiel15%claude-seo: 0%. Le gap concurrentiel est le meilleur argument de vente.
GEO / AI Search10%claude-seo: 10%. Identique, differenciateur SLASHR.
Local (conditionnel)5%claude-seo: 0% (separe). Poids faible sauf si business local.

Difference cle avec claude-seo : le /audit SLASHR ne produit pas juste un diagnostic technique, il repond a la question "est-ce que ce prospect a un potentiel Search suffisant pour justifier une proposition ?". C'est un enrichissement du /qualify, pas un remplacement.

Format d'une extension

extensions/dataforseo/
├── extension.md          # Manifest : nom, version, MCP server requis, capabilities
├── agent.md              # Spec du subagent (outils autorises, output format)
└── field-config.json     # Filtrage des champs API (optionnel)

Principe : le preflight_check.py detecte quels MCP servers sont actifs. Chaque passe adapte ses modules en fonction. Pas de code conditionnel dans les agents : si l'extension est absente, le subagent n'est simplement pas spawne.

Resume : ce que SLASHR v12 prend de claude-seo

Parallelisme des subagents

Pass 1 : 5 collecteurs paralleles. Pass 3 : 6 writers paralleles. Temps /prepare divise par 2.

Systeme d'extensions

Chaque source de donnees = 1 dossier autonome. Ajouter Semrush ou Ahrefs demain = 0 modif du core.

References on-demand

Les references techniques (CWV, E-E-A-T, GEO) sont chargees par les subagents qui en ont besoin, pas dans le contexte initial.

/audit : diagnostic rapide

Comme /seo audit, mais calibre pour le closing. Score SEO pondre par potentiel de marche + gap concurrentiel.

Ce que SLASHR v12 garde et que claude-seo n'a pas

Checkpoints humains

2 arrets obligatoires. Le closer valide la strategie et le plan narratif. Pas d'autonomie totale sur les delivrables client.

HTML interactif sur-mesure

30 composants CSS, design system, simulateur ROI, board-ready A4. Pas du markdown generique.

54 regles de validation

4 layers automatisees. La rigueur de validation reste superieure a tout ce qui existe dans claude-seo.

Boucle de retroaction

/debrief enrichi : auto-analyse, pattern matching, injection dans les futurs /prepare. Le systeme apprend.
