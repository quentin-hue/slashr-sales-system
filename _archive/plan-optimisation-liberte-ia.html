Chaque spec du systeme tombe dans une de ces 3 categories :

OBJECTIF (le QUOI)

Ce que le prospect doit ressentir/comprendre. Ex: "Le prospect comprend pourquoi il perd du trafic". L'IA a besoin de ca.

CONTRAINTE (les INTERDITS)

Les garde-fous non-negociables. Ex: "Zero pression commerciale", "Evidence chain". L'IA a besoin de ca.

METHODE (le COMMENT) — a supprimer

La facon de faire. Ex: "Ouvrir par un gap concurrentiel", "Utiliser un bar-chart pour le benchmark", "3 cards pricing en grid". L'IA n'a PAS besoin de ca. C'est la que se joue la liberation.

2. Pass 1 : de "10 modules sequentiels" a "collecter et diagnostiquer"

Probleme actuel

Pass 1 est un script de 350+ lignes avec 10 modules numerotes, des sous-etapes (1.1b, 4b, 4c, 4d), des Budget Checkpoints, et des conditions en cascade. L'IA suit un plan de vol au lieu de raisonner.

Transformation proposee

ElementAvant (deterministe)Apres (libre + contraintes)
Collecte 10 modules dans un ordre fixe, avec conditions explicites pour chaque module conditionnel FAIT 5 subagents paralleles. L'orchestrateur les lance tous, chacun gere son perimetre.
Modules conditionnels "Activer si SEA_SIGNAL != ABSENT", "Activer si Shopping features sur 3+ queries" A FAIRE L'IA decide quels modules activer en fonction des donnees collectees. Contrainte : budget API max, pas de module sans signal dans les sources.
Diagnostic S7 : scorer 7 forces 0-5, classifier PRIMARY/SECONDARY/DEFERRED FAIT Diagnostic libre : contrainte principale + 2 leviers max + confiance. L'IA raisonne, pas elle ne remplit une grille.
SDB Template fixe avec champs obligatoires (DOMAINE_PRINCIPAL, COMPETITIVE_GAP, INTENT_MARKET_MAP, etc.) A FAIRE Le SDB garde sa structure thin (obligatoire pour la performance), mais les sections sont libres. Seuls les champs critiques sont imposes (domaine, sources, evidence log).
ROI Methodologie implicite, varie d'un deal a l'autre A FAIRE 3 methodes documentees dans context/references/roi-methodology.md. L'IA choisit la plus adaptee et l'applique de facon reproductible.

Fichiers impactes

  • agents/prepare-pass1.md : simplifier de 350 lignes a ~150 lignes (objectif + contraintes, pas pas-a-pas)
  • Creer context/references/roi-methodology.md

3. Pass 2 : de "menu de hooks" a "convaincre ce decideur"

Probleme actuel

Pass 2 liste 6 types de hooks, impose une cascade 3 couches, definit des arcs types, fixe l'ordre des sections. L'IA choisit dans un menu au lieu d'inventer.

Transformation proposee

ElementAvantApres
Hook 6 hooks listes (gap, verbatim, paradoxe, cout inaction, opportunite, ancrage identitaire) A FAIRE Objectif : "L'ouverture est le fait le plus frappant pour CE prospect." Contrainte : cascade QUOI/POURQUOI/PREUVE. L'IA invente le hook.
Arc narratif Arcs types listes dans le flux principal (biais d'ancrage) A FAIRE Deplacer les arcs types dans context/references/arcs-narratifs.md comme inspiration, pas dans le flux. Le flux dit : "Emmener le decideur du constat a la conviction."
Sections Diagnostic "L'agent cree les sections qu'il veut" (bien en theorie, mais encadre par trop d'exemples) A FAIRE Garder la liberte. Supprimer les exemples du flux. Contrainte unique : max 5 slides, alternance data/interpretation.
Cas clients Selection par match sectoriel + metrique cle A FAIRE Selection par pertinence de l'angle strategique (primaire), puis secteur (secondaire). Un cas e-commerce peut illustrer un deal B2B si le levier est le meme.
Ton Identique sur tous les deals ("partenaire strategique") A FAIRE Ajouter un champ TONE_PROFILE dans le SDB : DIRECT (defaut) | PEDAGOGIQUE | PROVOCATEUR | TECHNIQUE. Le closer peut moduler au Checkpoint 1.
Onglets 4 obligatoires + 2 conditionnels, ordre fixe A FAIRE 4 obligatoires minimum. L'IA peut proposer un onglet custom au Checkpoint 2 (ex: "Migration" pour une refonte lourde). Le closer valide ou refuse.

Fichiers impactes

  • agents/prepare-pass2.md : reecrire en objectif + contraintes
  • Creer context/references/arcs-narratifs.md (exemples deplacer ici)

4. Pass 3 : de "mapper composant par composant" a "mettre en scene"

Probleme actuel

Pass 3 a un mapping composant-par-role-narratif explicite. L'IA suit une table de correspondance au lieu de faire du design.

Transformation proposee

ElementAvantApres
Choix composants Table de mapping (donnee quantitative → bar-chart, comparaison → table, etc.) A FAIRE L'IA choisit les composants du kit. Contraintes : rythme visuel (pas 2 charts consecutifs), max 2 composants visuels par slide, le kit CSS reste la reference.
Rythme visuel Regles explicites (alternance data/highlight) A FAIRE Objectif : "Le prospect ne decroche jamais." Contrainte : pas 2 blocs data sans interpretation. L'IA gere le rythme.
Simulateur ROI Structure JS fixe avec sliders specifiques A FAIRE L'IA genere le JS custom adapte au deal. Si le deal n'a pas de CVR mesure, pas de slider CVR. Les inputs du simulateur = les hypotheses reelles du SDB, pas un template.

5. Validation : de 54 regles a "garde-fous + liberte"

Probleme actuel

54 regles sur 4 layers. Certaines sont des vrais garde-fous (zero S7, evidence chain). D'autres sont des preferences de mise en forme deguisees en regles (regle 8 : "alternance data/highlight-box").

Transformation proposee

Reclassifier les 54 regles en 2 categories :

CategorieNombre estimeExemplesComportement
HARD (non-negociable) ~25 Zero jargon interne dans le HTML, evidence chain, accents obligatoires, pas de tiret cadratin, pas de jours/TJM visibles, coherence levier Phase 1 ↔ Phase 2 PASS/FAIL automatique. L'IA n'a aucune liberte.
SOFT (recommandation) ~29 Alternance data/interpretation, nombre de slides par onglet, structure FAQ, presence de "Ce que cela implique" WARNING informatif. L'IA peut violer une regle soft si elle justifie pourquoi (ex: "Pas d'alternance data/highlight dans cet onglet car il n'y a que 2 sections").

Fichiers impactes

  • context/validation_rules.md : reclassifier chaque regle HARD/SOFT
  • tools/validate_proposal.py : ajouter un flag --strict (HARD only) vs --full (HARD + SOFT)

6. Prepare-context.md : de "bundle de 8 fichiers" a "brief d'objectifs"

Probleme actuel

agents/prepare-context.md est un copie-colle condense de 8 fichiers (~200+ lignes). L'IA le lit et retient des instructions, pas des objectifs.

Transformation proposee

Reecrire en 3 blocs :

## 1. Ton objectif
Generer une proposition HTML qui convainc CE decideur d'investir dans sa visibilite Search.
Pas un template. Pas une plaquette. Un argumentaire sur-mesure, base sur les donnees de CE prospect.

## 2. Tes contraintes (non-negociables)
- Evidence chain : chaque chiffre source
- Zero jargon interne (diagnostic, leviers, confiance = internes)
- Accents francais obligatoires
- Zero pression commerciale / dramatisation
- Lexique interdit : [liste]
- Max 3 leviers actifs
- Pricing : 1 recommandation, pas 3 choix

## 3. Tes outils
- Kit composants : `templates/proposal-kit.html` (30 composants)
- Design system : `context/design_system.md`
- Cas clients : `context/case_studies.md`
- Pricing : `context/pricing_rules.md`
- References on-demand : `context/references/` (charger si besoin)

~50 lignes au lieu de 200+. L'IA sait quoi faire, pourquoi, et avec quels outils. Le comment est libre.
